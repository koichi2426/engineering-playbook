# 2. コミュニケーション (Communication)

このドキュメントは、プロジェクトにおける効率的なコミュニケーションのためのツールの使い分けとルールを定義します。

## 1. 基本方針 (Basic Policy)

1.  **オープン (Open):** DM（ダイレクトメッセージ）での議論は避け、原則としてパブリックなチャンネルやIssueで行います。
2.  **非同期 (Asynchronous):** リアルタイム性を強要せず、各自が集中できる時間を尊重します。
3.  **記録 (Stock):** 将来の参照やAIによる学習のため、フロー（流れる情報）ではなくストック（蓄積される情報）を重視します。**特に GitHub Issues を情報のハブとします。**

## 2. ツールの使い分け (Tool Usage)

### A. GitHub Issues (ストック型の議論・タスク管理)

**最も重要な情報の蓄積場所です。**

* **用途:**
    * 機能要望、仕様の提案、設計に関する議論
    * バグの報告と管理
    * タスクの進捗管理
* **ルール:**
    * 議論の**背景**と**決定事項**を必ず記録します。
    * Copilot Workspace は Issue を起点に動作するため、AIが理解できるよう明確に記述します。
    * Slackで始まった議論でも、仕様が固まった時点や作業が発生した時点で Issue を作成し、リンクを共有します。

### B. Slack / Teams (フロー型の議論・日常連絡)

* **用途:**
    * 日常的な連絡、簡単な質問、相談
    * 緊急の連絡
    * 雑談
* **ルール:**
    * テキストでの議論が10分以上続きそうな場合や、複雑なテーマの場合は、ビデオ会議かGitHub Issuesでの議論に切り替えます。
    * **決定事項は、必ず GitHub Issues に転記します。**

### C. GitHub Pull Requests (PR) (コードに関する議論)

* **用途:**
    * 実装されたコードに関する具体的なレビュー、質問、改善提案。
* **ルール:**
    * なぜこの変更が必要なのか（`What` & `Why`）は、PRの説明欄（または関連Issue）に記述します。
    * レビューコメントは、コードそのもの（`How`）に焦点を当てます。

### D. ビデオ会議 (Meet / Zoom / etc.)

* **用途:**
    * テキストでは伝達が困難な、複雑な設計のブレインストーミングやすり合わせ。
* **ルール:**
    * 会議の目的を事前にアジェンダで共有します。
    * 会議後は、**決定事項と議論の概要を必ず GitHub Issues に要約**して記録します。

## 3. メンションと通知 (Mentions)

* `@here` / `@channel` (Slack等) の使用は、全員に即時通知が必要な緊急時のみに限定します。
* GitHub でレビューを依頼する際は、コメントで `@` メンションを送るだけでなく、必ず `Reviewers` 機能で担当者を指定してください。
